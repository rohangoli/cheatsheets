Docker Architecture

Docker uses a client-server architecture to build, ship, and run containerized applications with lightweight isolation.

üöÄ Applications & Services
Containerized applications running in isolated environments
üì¶ Containers
Runtime instances of Docker images with own filesystem, network, and process space
üñºÔ∏è Docker Images
Read-only templates with instructions for creating containers
‚öôÔ∏è Docker Engine (dockerd)
Server daemon managing containers, images, networks, and volumes
üê≥ Docker Client (docker CLI)
Command-line interface communicating with Docker daemon via REST API
üíª Host Operating System
Linux kernel with namespaces and cgroups for isolation
Key Concepts

Images: Immutable templates containing application code, dependencies, and configuration

Containers: Running instances of images, isolated but sharing host kernel

Registry: Storage and distribution system for Docker images (Docker Hub, private registries)

Dockerfile: Text file with instructions to build Docker images

Docker Swarm Mode

What is Docker Swarm?

Docker Swarm is native clustering and orchestration for Docker. It turns multiple Docker hosts into a single virtual host with automatic load balancing, service discovery, and rolling updates.

Key Concepts: Manager nodes (control plane), Worker nodes (run tasks), Services (desired state), Tasks (container instances)

Initialize Swarm
docker swarm init
docker swarm init --advertise-addr 192.168.1.100
Creates swarm cluster on current node as manager
Join Swarm (Worker)
docker swarm join --token SWMTKN-... 192.168.1.100:2377
Joins worker node to existing swarm (use token from init)
Join Swarm (Manager)
docker swarm join-token manager
docker swarm join --token SWMTKN-... 192.168.1.100:2377
Gets manager token and joins as manager node
Get Join Token
docker swarm join-token worker
docker swarm join-token manager
Displays join command with token for workers/managers
List Nodes
docker node ls
Shows all nodes in swarm with status and role
Inspect Node
docker node inspect node-name
Displays detailed node information
Promote/Demote Node
docker node promote node-name
docker node demote node-name
Changes node between worker and manager role
Update Node Availability
docker node update --availability drain node-name
docker node update --availability active node-name
Sets node to drain (evacuate tasks) or active
Remove Node
docker node rm node-name
docker node rm -f node-name
Removes node from swarm (node must be down)
Leave Swarm
docker swarm leave
docker swarm leave --force
Removes current node from swarm (--force for manager)
Swarm Info
docker info
Shows swarm status and configuration
Update Swarm Config
docker swarm update --autolock=true
Updates swarm configuration (autolock protects encryption keys)

Swarm Architecture

1
Manager Nodes
Orchestration, scheduling, cluster management
2
Worker Nodes
Execute tasks assigned by managers
3
Services
Define desired state (replicas, image, etc.)
4
Tasks
Running container instances
Swarm Best Practices

‚úì Use odd number of manager nodes (3, 5, 7) for high availability

‚úì Don't exceed 7 manager nodes (Raft consensus overhead)

‚úì Place managers in different availability zones

‚úì Use drain availability for maintenance without disruption

‚úì Enable autolock to protect encryption keys

Docker Services (Swarm)

Services vs Containers

Services are the desired state for your application in swarm mode. Swarm automatically creates and manages containers (tasks) to match the service definition, with features like scaling, rolling updates, and self-healing.

Create Service
docker service create --name web \
--replicas 3 -p 8080:80 nginx
Creates service with 3 replicas, publishes port 8080
List Services
docker service ls
Shows all services with replica count and status
Service Details
docker service ps service-name
Lists all tasks (containers) for service with node placement
Inspect Service
docker service inspect service-name
docker service inspect --pretty service-name
Displays detailed service configuration (--pretty for readable)
Scale Service
docker service scale web=5
docker service scale web=5 api=3
Changes number of replicas (can scale multiple services)
Update Service
docker service update --image nginx:1.21 web
docker service update --env-add ENV=prod web
Updates service with rolling update (new image, env vars, etc.)
Rollback Service
docker service rollback service-name
Rolls back to previous service configuration
Remove Service
docker service rm service-name
Deletes service and all its tasks
Service Logs
docker service logs service-name
docker service logs -f --tail 100 service-name
Views logs from all service tasks
Update Constraints
docker service update \
--constraint-add node.role==worker web
Adds placement constraint (only run on worker nodes)
Resource Limits
docker service update \
--limit-cpu 1 --limit-memory 512M \
--reserve-cpu 0.5 --reserve-memory 256M web
Sets CPU/memory limits and reservations
Update Parallelism
docker service update \
--update-parallelism 2 \
--update-delay 10s web
Updates 2 tasks at a time with 10s delay between
Global Service
docker service create --mode global \
--name monitor monitoring-agent
Runs one task on every node (useful for monitoring)
Add Volume
docker service create \
--mount type=volume,source=data,target=/data app
Mounts volume to service tasks
Service Networks
docker service create \
--network mynetwork web
Connects service to overlay network
Publish Ports
docker service create \
--publish published=8080,target=80,mode=ingress web
Publishes port with ingress load balancing
Service Update Strategies

Rolling Update (default): Gradually replaces tasks one by one or in batches

Update Order: stop-first (default) or start-first

Failure Action: pause (default), continue, or rollback

Docker Stack (Swarm)

What is Docker Stack?

Docker Stack allows you to deploy multi-service applications to a swarm using a Compose file. It's the production-ready way to manage complex applications in swarm mode with declarative configuration.

Deploy Stack
docker stack deploy -c docker-compose.yml mystack
Deploys or updates stack from compose file
List Stacks
docker stack ls
Shows all deployed stacks
Stack Services
docker stack services mystack
Lists all services in stack
Stack Tasks
docker stack ps mystack
Shows all tasks (containers) in stack with node placement
Remove Stack
docker stack rm mystack
Removes stack and all its services, networks, volumes
Multiple Compose Files
docker stack deploy \
-c docker-compose.yml \
-c docker-compose.prod.yml mystack
Deploys stack with merged compose files (overrides)
Stack Compose File Example

version: '3.8'
services:
  web:
    image: nginx:alpine
    deploy:
      replicas: 3
      update_config:
        parallelism: 1
        delay: 10s
      restart_policy:
        condition: on-failure
      placement:
        constraints:
          - node.role == worker
    ports:
      - "8080:80"
    networks:
      - webnet
networks:
  webnet:
    driver: overlay

Stack vs Compose Differences

‚Ä¢ Stack ignores build: directive (use pre-built images)

‚Ä¢ Use deploy: section for swarm-specific configs

‚Ä¢ Networks are automatically created as overlay type

‚Ä¢ Volume drivers must be available on all nodes

‚Ä¢ docker-compose up works on single host; docker stack deploy for swarm

Docker Secrets (Swarm)

What are Docker Secrets?

Docker Secrets provide secure storage and transmission of sensitive data (passwords, API keys, certificates) to services in swarm mode. Secrets are encrypted at rest and in transit, only available to services explicitly granted access.

Storage: In-memory tmpfs at /run/secrets/ inside containers (never written to disk)

Create Secret (stdin)
echo "mysecretpassword" | docker secret create db_password -
Creates secret from stdin (dash indicates stdin)
Create Secret (file)
docker secret create db_password ./password.txt
Creates secret from file contents
List Secrets
docker secret ls
Shows all secrets (values are never displayed)
Inspect Secret
docker secret inspect db_password
Shows metadata (not the actual secret value)
Remove Secret
docker secret rm db_password
Deletes secret (must not be in use by any service)
Service with Secret
docker service create \
--name db \
--secret db_password \
postgres
Creates service with access to secret
Custom Secret Path
docker service create \
--secret source=db_password,target=password.txt \
myapp
Mounts secret at custom path (/run/secrets/password.txt)
Secret with Permissions
docker service create \
--secret source=cert,target=app.crt,mode=0400 \
web
Sets file permissions on secret (0400 = read-only for owner)
Update Service Secret
docker service update \
--secret-rm old_password \
--secret-add new_password \
myservice
Replaces secret in running service (triggers rolling update)
Read Secret in Container
cat /run/secrets/db_password
DB_PASS=$(cat /run/secrets/db_password)
Applications read secrets from /run/secrets/ directory
Stack with Secrets
services:
  db:
    image: postgres
    secrets:
      - db_password
secrets:
  db_password:
    external: true
References secret in stack compose file (must exist)
Create Secret in Stack
secrets:
  db_password:
    file: ./password.txt
Creates secret from file during stack deployment
Secret Best Practices

‚úì Never commit secrets to version control

‚úì Use external secrets in production (managed outside stack file)

‚úì Rotate secrets regularly by creating new ones and updating services

‚úì Use principle of least privilege (only grant access to services that need it)

‚úì Secrets are immutable - create new secret to update value

‚úì Combine secrets with environment variables: POSTGRES_PASSWORD_FILE=/run/secrets/db_password

Secrets vs Environment Variables

Secrets: Encrypted, not visible in docker inspect, in-memory only, swarm-only

Environment Variables: Plaintext, visible in inspect, stored in config, work everywhere

Use secrets for sensitive data in production swarm deployments

Best Practices

Dockerfile Optimization

‚úì Use specific image tags, not latest (e.g., nginx:1.21-alpine)

‚úì Combine RUN commands to reduce layers: RUN apt-get update && apt-get install -y pkg

‚úì Order instructions from least to most frequently changed

‚úì Use .dockerignore to exclude unnecessary files

‚úì Use multi-stage builds for smaller production images

‚úì Run as non-root user for security

Container Management

‚úì Use meaningful container names (--name)

‚úì Set restart policy for production (--restart=unless-stopped)

‚úì Define resource limits to prevent resource exhaustion

‚úì Use health checks to monitor container status

‚úì Keep containers ephemeral - store data in volumes

‚úì One process per container (separation of concerns)

Production Deployment

‚úì Use Docker Compose for multi-container apps

‚úì Use Swarm or Kubernetes for orchestration

‚úì Implement centralized logging (ELK stack, Splunk)

‚úì Monitor containers with Prometheus, Grafana, or similar

‚úì Use overlay networks for swarm communication

‚úì Implement CI/CD pipelines for automated deployments

‚úì Regular security scanning and updates

Common Pitfalls

‚ö† Not cleaning up unused resources (images, volumes, networks)

‚ö† Storing secrets in images or environment variables

‚ö† Running containers as root user

‚ö† Not setting resource limits (CPU, memory)

‚ö† Using latest tag in production

‚ö† Not implementing health checks

‚ö† Ignoring container logs
